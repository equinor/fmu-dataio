Module for DataIO class.

The metadata spec is documented as a JSON schema, stored under schema/.

The processing is based on handling first level keys which are

  • Scalar special:

    $schema      |
version      |     hard set in code, corresponding to schema version
source       |

  • class - determined by datatype, inferred

  • Nested attributes:

    file         - information about the data object origin as file on disk
tracklog     - events recorded on these data
data         - about the data (see class). Inferred from input, data + fmuconfig
display      - Deduced mostly from fmuconfig (TODO: issue on wait)
fmu          - Deduced from fmuconfig and ERT
access       - Govern permissions for exported data, infer from fmuconfig or args
masterdata   - Static, infer from fmuconfig

Class for exporting data with rich metadata in FMU.

This class sets up the general metadata content to be applied in export. The idea is that one ExportData instance can be re-used for several similar export() jobs. For example:

edata = dataio.ExportData(
    config=CFG,
    content="depth",
    unit="m",
    vertical_domain={"depth": "msl"},
    timedata=None,
    is_prediction=True,
    is_observation=False,
    tagname="faultlines",
    workflow="rms structural model",
)

for name in ["TopOne", TopTwo", "TopThree"]:
    poly = xtgeo.polygons_from_roxar(PRJ, hname, POL_FOLDER)
    out = ed.export(poly, name=name)

Some keyword settings like name, tagname etc can be set in both the ExportData instance and directly in the export() function, to provide flexibility for different use cases. If both are set, the export() setting will win.

export(obj, subfolder=None, forcefolder=None, verbosity=None, name=None, unit=None, parent=None, tagname=None, display_name=None, description=None, include_index=False, **kwargs)[source]ÔÉÅ

Export data objects of ‚Äòknown‚Äô type to FMU storage solution with metadata.

A ‚Äòknown‚Äô type is a type which is known to fmu-dataio. Examples of such known types are XTGeo objects (e.g. a RegularSurface), a Pandas Dataframe, etc.

This function will also collect the data spesific class metadata. For ‚Äúclassic‚Äù files, the metadata will be stored i a YAML file with same name stem as the data, but with a . in front and ‚Äúyml‚Äù and suffix, e.g.:

top_volantis--depth.gri
.top_volantis--depth.gri.yml

For formats supporting embedded metadata, other solutions may be used.

Parameters

  • obj (Any) ‚Äì XTGeo instance or a Pandas Dataframe instance (more to be supported).

  • subfolder (Optional[str]) ‚Äì Optional subfolder below standard level to export to.

  • verbosity (Optional[str]) ‚Äì Verbosity level of logging messages. If not spesified, the verbosity level from the instance will be used.

  • name (Optional[str]) ‚Äì Override any setting in class initialization. The name of the object. If not set it is tried to be inferred from the xtgeo/pandas/‚Ä¶ object. The name is then checked towards the stratigraphy list, and name is replaced with official stratigraphic name if found in static metadata stratigraphy. For example, if ‚ÄúTopValysar‚Äù is the model name and the actual name is ‚ÄúValysar Top Fm.‚Äù that latter name will be used.

  • unit (Optional[str]) ‚Äì Is the unit of the exported item(s), e.g. ‚Äúm‚Äù or ‚Äúfraction‚Äù.

  • tagname (Optional[str]) ‚Äì Override any setting in class initialization. This is a short tag description which will be a part of file name and data.tagname.

  • parent (Optional[str]) ‚Äì Override any setting in class initialization. This key is required for datatype GridProperty, and refers to the name of the grid geometry.

  • description (Optional[str]) ‚Äì Override any setting in class initialization. A multiline description of the data.

  • display_name (Optional[str]) ‚Äì Override any setting in class initialization. Set name for clients to use when visualizing. Defaults to name if not given.

  • include_index (Optional[bool]) ‚Äì For Pandas table output. If True, then the index column will be included. For backward compatibilty, index also supported.

Returns

full path to exported item.

Export the case metadata to file, e.g. when in an ERT run.

This will be the configuration file and output the data necessary to generate a general case ID, typically done as a hook workflow in ERT or similar.

Parameters

  • casename ‚Äì Name of case (run), e.g. ‚Äòmycase‚Äô

  • caseuser ‚Äì The username fro the case, e.g. the <USER> in ERT

  • restart_from ‚Äì The uid of iteration from whci to run from

  • description ‚Äì A free-form description for case
